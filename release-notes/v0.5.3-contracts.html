V5.3 contracts release — rotationNewWallet 160-bit range-check (F2) ​

Renamed in v0.6.0 — content reflects v0.5.x state; subsequent versions use zkqes nomenclature.

V5.3 is an in-place amendment to the V5.2 contract (NOT a new contract). The V5.2 amendment design picks up F2 (rotationNewWallet 160-bit range-check) on the contract side; F1 (subject-serial OID-anchor) and F3 (walletSecret↔msgSender doc) are circuits-eng's track. Why V5.3 (contract-side) ​

V5.2's contract review surfaced three findings; F2 is the contract-side one. The V5.2 rotateWallet() flow contains a silent-truncation surface at the public-signal unpack:

solidity
address newWallet = address(uint160(sig.rotationNewWallet));

sig.rotationNewWallet is a public-signal slot (uint256, BN254 field-element-shaped). Without a range-check, a malicious prover can submit sig.rotationNewWallet = (high_bits << 160) | uint160(prover_addr) — the cast yields prover_addr, the downstream newWallet != msg.sender check passes (prover prank as prover_addr), and the proof's slot 17 carries an arbitrary 254-bit field element that the proof commits to but the contract never validates against the cast result.

This is a defense-in-depth gap rather than an immediate exploit (the attacker still needs to satisfy the Groth16 pairing equation against the published verifier, and the V5.3 in-circuit Num2Bits(160) constraint will close the same hole at the circuit layer once circuits-eng's track lands). But the load-bearing-on-each-other discipline (gate at both circuit and contract) means either layer can be refactored without re-introducing the truncation surface.

Contract-side change (one diff) ​

In QKBRegistryV5_2.rotateWallet(), between the existing WrongMode mode gate and the Groth16 verify call:

solidity
// V5.3 F2: rotationNewWallet 160-bit range check.
if (sig.rotationNewWallet != uint256(uint160(sig.rotationNewWallet))) {
    revert InvalidNewWallet();
}

Insertion is at the earliest fail-fast position so malformed proofs don't pay the ~400K Groth16 verifier cost. The check uses the existing InvalidNewWallet() error declaration from V5.2 — semantics generalize from V5.2's "newWallet == 0 || == oldWallet || != msg.sender" to "+ bits 160..255 nonzero" without selector change.

What about register()? ​

Not added — and intentionally so. The V5.2 register() flow already enforces an equivalent bound via the existing WrongRegisterModeNoOp gate:

solidity
if (sig.rotationNewWallet != uint256(uint160(msg.sender))) {
    revert WrongRegisterModeNoOp();
}

uint256(uint160(msg.sender)) is bounded by 2^160 - 1 by construction (any uint160 cast produces a value ≤ uint160.max). For this gate to PASS, sig.rotationNewWallet MUST equal that value, hence MUST be < 2^160. F2 in register() would be structurally redundant — the existing gate already enforces the 160-bit bound as a side effect of the equality check.

Adding F2 to register() for spec-uniformity over structural redundancy was considered and rejected:

  • ~60 gas extra on the high-volume register path (vs zero benefit; the gate would never fire / never catch anything)
  • Future readers see two "same" gates and assume something subtle is going on
  • If WrongRegisterModeNoOp is ever refactored away, the resulting bug ("register-mode rotateNewWallet unbounded") would be in plain sight + caught by its own test coverage anyway, NOT silenced by a second gate

The contract-side F2 scope is therefore rotateWallet-only. Spec §F2.2's "add to both flows" guidance is being amended to reflect this in the v0.2 spec refresh.

Public-signal layout ​

No changes. V5.3 keeps the V5.2 frozen layout exactly (22 signals, slots 0..21). F2's contract-side check operates on slot 17 (rotationNewWallet) without altering its position or shape. F1 + F3 (circuits-eng's track) add private witness inputs but no new public outputs.

What stayed the same ​

  • IQKBRegistry interface — selectors and return types byte-identical V5/V5.1/V5.2/V5.3.
  • Storage layout — 7 slots, byte-identical to V5.2.
  • Constructor signature — (verifier, admin, trustRoot, policyRoot) parity-equivalent.
  • register() flow — F2 not added (structurally redundant per §"What about register()?"); V5.2 register-path gas measurements unchanged.
  • Public-signal layout — 22 signals, frozen per V5.2 §"Public-signal layout".
  • All non-InvalidNewWallet custom error selectors — byte-identical.
  • rotateWallet() rotation auth model — ECDSA sig from oldWallet's privkey over the rotation payload (chainid + registry addr + fingerprint + newWallet); F2 fires before the auth check.
  • script/DeployV5_2.s.sol — reusable for V5.3 deploy with same env layout; only GROTH16_VERIFIER_ADDR changes.
  • First-claim nullifier semantics — V5.1 invariant 4 (nullifierOf write-once on first-claim, migrates atomically on rotateWallet) preserved verbatim.

Migration guidance ​

For SDK consumers (web, integrators) ​

  1. Bump dependency: @qkb/sdk@0.5.3-pre.
  2. No ABI re-bind needed. All selectors byte-identical V5.2 → V5.3. Continue using qkbRegistryV5_2Abi.
  3. No revert-classifier change. InvalidNewWallet's selector is preserved; the new high-bits-set trigger condition reverts with the same selector existing classifiers already handle.
  4. For witness builders: pick up V5.3 witness-builder additions (F1's subjectSerialOidOffsetInTbs private input) when web-eng T5 lands. Public-output layout still 22 — no calldata change.

For deployer-tooling ​

  1. Use script/DeployV5_2.s.sol (unchanged) for the V5.3 deploy.
  2. Point GROTH16_VERIFIER_ADDR at the V5.3 ceremonied verifier (post-Phase-B V5.3 multi-contributor ceremony at ceremony/v5_3/).
  3. Same env layout otherwise: PRIVATE_KEY, ADMIN_ADDRESS, INITIAL_TRUST_ROOT, INITIAL_POLICY_ROOT, MINT_DEADLINE, optional CHAIN_LABEL.

For holders ​

V5.2 → V5.3 is a fresh deploy, not an upgrade. The registry contract itself (QKBRegistryV5_2.sol) is non-upgradeable; the F2 gate is in-place source amendment; the bytecode that rolls is the new bytecode of the V5.3 deploy + a new V5.3-ceremonied groth16Verifier immutable. Existing V5.2 holders re-register against the new V5.3 registry via the standard fresh-QES flow.

The two registries (V5.2 and V5.3) can coexist on-chain during the transition window; downstream consumers point at one or the other via deployment-fixture configuration (fixtures/contracts/{sepolia,mainnet}.json).
